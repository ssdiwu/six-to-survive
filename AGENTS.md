# Agent 工作规范

## 项目定位

《Nine to Survive · 撑到六点》是浏览器 2D 弹幕 Roguelike 独立小游戏，用于豆包 Seed-Evolving 模型开发者测评。核心目标：让 AI Coding 模型（在 TRAE 中调用 Evolving API）一次成型交付完整可玩的单文件网页游戏。

当前处于 **v0.1 规格基线**：PRD 已完成，`index.html` 仍是待开发的 Canvas 骨架。不要把文档中的目标能力误判为已经实现。

## 权威顺序

协作规则以本文档为准；理解项目内容时按以下顺序读取：

1. `README.md`：项目入口、当前状态、运行方式、目录地图
2. `doc/README.md`：文档地图、阅读顺序、权威与归档边界
3. `doc/00-产品与原则/PRD.md`：产品需求、玩法规则、数值边界
4. `index.html`：当前实现真相源；文档中的目标不代表代码已经实现
5. 目标代码目录自己的 `README.md`，再读具体源码

## 工程约束

- **单文件 HTML 交付**：最终产物收敛到 `index.html`，内联 CSS + JS，零构建、零 npm 依赖
- **Canvas 2D 渲染**：不使用 three.js / PixiJS / WebGL；游戏离线运行不得依赖外部图片、音频或字体，可选 Google Fonts CDN 只能作为增强且必须有系统字体兜底
- **表驱动**：敌人、词条、波次、技能配置全部是 JS 常量对象/数组，便于 AI 生成、调参和扩展
- **开发期目录**：`scripts/` `styles/` `assets/` 仅用于开发调试，不进最终交付
- **无后端、无存档、无联网**：纯前端、纯本地

## 代码风格

- Vanilla JS（ES2020+），不用 TypeScript、不用框架
- 代码分块清晰：`// === 配置 ===` / `// === 状态 ===` / `// === 引擎 ===` / `// === 渲染 ===` / `// === 主循环 ===`
- 函数保持小而纯，核心引擎（伤害/词条/波次）写成纯函数便于 console 单测
- 注释用中文，写"为什么"不写"是什么"

## 工程纪律

- **三次原则**：同一种变化出现第三次再抽象，少量重复优于提前设计通用层
- **删除测试**：判断模块是否值得存在时，想象删掉它；复杂度消失则删，复杂度扩散到调用处则保留
- **接缝纪律**：只有出现两个以上真实实现时才引入接口；单实现 adapter 属于假设接缝
- **函数粒度**：函数控制在 100 行以内；超出时按职责拆分
- **测试看行为**：通过公共入口验证行为，不测试内部实现；mock 只放在浏览器 API 等系统边界
- **先建反馈环再调 bug**：先建立可重复的 console 断言、浏览器复现或 headless 检查，再修改代码
- **插桩打 tag**：临时调试日志使用唯一前缀（如 `[DEBUG-a4f2]`），验证后统一删除

## 验证方式

- 当前骨架：启动 `python3 -m http.server 8000`，打开 `http://localhost:8000`，确认页面渲染且控制台无报错
- 功能实现后：完整跑通 9:00→18:00，验证胜利、失败和 Boss 三阶段
- 核心纯函数（伤害计算、词条触发、波次推进）用浏览器 console 断言验证
- 项目不引入自动化测试框架；单元层使用 console 断言，集成与端到端层使用浏览器手测，并在交付时说明缺失层级

## 文档同步

- 改代码前先读目标目录及上下级 `README.md`
- 含代码的目录必须有 `README.md`；目录无实际内容时惰性创建
- 职责、结构、入口、运行方式或约定变化时，同步更新根 `README.md` 与 `doc/README.md`
- 稳定项目术语写入 `doc/术语表.md`；只有同时满足难逆转、无上下文会困惑、有真实权衡的决策才写入 `doc/决策档案/`

## Git 规范

- 每次提交只做一件事，精准暂存相关文件
- 提交信息使用 Conventional Commits：`type: 中文描述`
- 提交前检查 `CHANGELOG.md` 的 `Unreleased` 是否需要记录玩家可感知变更
- 禁止向 `main` 分支执行强制推送

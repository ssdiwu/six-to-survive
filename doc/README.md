# 文档地图

## 阅读顺序

1. `../README.md`：项目入口、当前状态、运行方式与目录地图
2. 本文件：文档分区、权威性与历史边界
3. `00-产品与原则/PRD.md`：产品目标、玩法规则、数值边界与待定项
4. `../index.html`：当前实现真相源；文档里的目标不代表已经实现
5. 目标代码目录自己的 `README.md`，再读具体源码

## 权威与边界

- `00-产品与原则/PRD.md` 是当前产品规格权威；其中标注为“待定”的内容尚未冻结
- `../index.html` 与后续代码是当前实现权威；实现状态以代码和验证结果为准
- `../AGENTS.md` 是 AI 协作与工程约束权威
- `../CHANGELOG.md` 只记录玩家可感知的版本变化，不承载路线图或内部计划
- 历史材料后续统一放入 `90-归档/`，只用于追溯，不作为当前方案依据

## 目录编号约定

- `00-产品与原则/`：PRD、玩法规则、体验原则
- `10-架构与运行/`：工程结构、渲染管线、输入处理、运行与验证
- `20-内容数据/`：敌人、词条、波次、技能等数值真相源
- `30-路线图/`：版本维度的迭代安排
- `40-版本实施方案/`：受路线图约束、带验收清单的版本落地方案
- `90-归档/`：已完成、过时或仅用于追溯的材料

只创建已有实际内容的目录，不为空规范补空目录。

## 知识沉淀

- 稳定领域词统一写入 `术语表.md`；新增前先对齐精确定义，不建空表
- 同时满足“难逆转、无上下文会困惑、有真实权衡”的决策写入 `决策档案/`
- 第一条决策落地时再创建 `决策档案/README.md` 索引，不提前创建空目录
- 可改的做法与避坑经验在真实问题出现后写入 `经验笔记.md`，不预建空文件
